# Building & Releasing KFCode

KFCode ships cross-platform binaries through [`dist`](https://github.com/axodotdev/cargo-dist)
(formerly `cargo-dist`). A single git tag drives the whole pipeline: compile on
native runners, package per platform, publish a GitHub Release, and update the
Homebrew tap.

## Release artifacts

Every tagged release produces:

| Platform | Target triple | Archive |
|----------|---------------|---------|
| Windows amd64 | `x86_64-pc-windows-msvc` | `.zip` |
| Linux amd64 | `x86_64-unknown-linux-gnu` | `.tar.gz` |
| macOS arm64 | `aarch64-apple-darwin` | `.tar.gz` |

Alongside the archives, each Release also includes:

- `kfcode.rb` — Homebrew formula (auto-pushed to the tap repo)
- `kfcode-cli-installer.sh` — shell one-line installer (`curl … | sh`)
- `kfcode-cli-installer.ps1` — PowerShell one-line installer (`irm … | iex`)
- `*.sha256` — a checksum next to every archive

> **Platform coverage:** only macOS **arm64** (Apple Silicon) is built. Intel Macs
> (`x86_64-apple-darwin`) are out of scope; `brew install` will fail on them.

The compiled binary is named `kfcode` and comes from the `kfcode-cli` crate. Inside
each archive it lives in a platform-named subdirectory (dist 0.32 default), e.g.
`kfcode-cli-aarch64-apple-darwin/kfcode`. The generated formula and installers
handle that path automatically, so users never deal with it directly.

## Configuration files

The pipeline is split between tool/platform-mandated locations and the human-facing
scripts directory.

### `dist-workspace.toml` (repo root)

The single source of truth for dist. Required at the repo root (or in `Cargo.toml`).
Key settings:

| Key | Value | Meaning |
|-----|-------|---------|
| `cargo-dist-version` | `0.32.0` | dist version pinned for reproducible CI |
| `ci` | `github` | generates a GitHub Actions workflow |
| `targets` | the 3 triples above | platforms to build |
| `installers` | `shell`, `powershell`, `homebrew` | artifacts to generate |
| `windows-archive` | `.zip` | archive format on Windows |
| `unix-archive` | `.tar.gz` | archive format elsewhere (dist default is `.tar.xz`) |
| `tap` | `dfbb/homebrew-tap` | repo the formula is pushed to |
| `formula` | `kfcode` | formula name → `brew install dfbb/tap/kfcode` |
| `publish-jobs` | `homebrew` | run the Homebrew publish job in CI |
| `hosting` | `github` | host Releases on GitHub |

Changing any of these requires regenerating the workflow (see *Maintenance*).

### `.github/workflows/release.yml` (generated)

Generated by dist; GitHub mandates this path. **Do not hand-edit** — it is produced
from `dist-workspace.toml` and verified in CI. Triggers:

- `push` of a tag matching `**[0-9]+.[0-9]+.[0-9]+*` (e.g. `v0.1.1`) → full build + publish
- `pull_request` → runs `dist plan` only, as a config sanity check (no publish)

Per-target runner mapping (computed by dist):

| Target | Runner |
|--------|--------|
| `aarch64-apple-darwin` | `macos-14` |
| `x86_64-pc-windows-msvc` | `windows-2022` |
| `x86_64-unknown-linux-gnu` | `ubuntu-22.04` |

Each target builds on its own native runner, avoiding cross-compilation. This matters
because KFCode pulls in native crates (`sqlx`+sqlite, `portable-pty`, `syntect`'s onig,
`reqwest`'s default TLS) that are painful to cross-compile.

### Cargo manifest changes

- Root `Cargo.toml`: a `[profile.dist]` (inherits `release`, `lto = "thin"`) and
  `repository` under `[workspace.package]`.
- `crates/kfcode-cli/Cargo.toml`: inherits `repository`, plus `description` and
  `homepage` (needed for a complete Homebrew formula).
- `crates/kfcode-lsp/Cargo.toml`: `[package.metadata.dist] dist = false` — its only
  `[[bin]]` is a test fixture, not a release artifact.

## Helper scripts (`scripts/build/`)

| Script | Purpose |
|--------|---------|
| `release.sh <version>` | Release entry point: validate, bump version, tag, push |
| `package-local.sh` | Build & package for the current host only (local smoke test) |
| `check-dist.sh` | Pre-release self-check: `dist plan` + `dist generate --check` |
| `README.md` | Short operator notes |

## One-time setup (before the first release)

Homebrew auto-publishing needs a tap repo and a cross-repo write token:

1. Create an empty GitHub repo **`dfbb/homebrew-tap`**. *(Done.)*
2. Generate a token with write access to that tap (fine-grained PAT, or classic PAT
   with `repo` / `contents:write`).
3. Add it to `dfbb/KFCode` → Settings → Secrets and variables → Actions as
   **`HOMEBREW_TAP_TOKEN`**. *(Done.)*

The default `GITHUB_TOKEN` cannot push to another repo, which is why this dedicated
secret is required.

> Never commit the token itself. `.gitignore` excludes `*token*.txt` and `*.token` as
> a safety net; the value lives only in the GitHub Actions secret.

## Installing dist locally

Only needed to run the helper scripts on your machine. CI installs its own pinned copy.

```bash
curl --proto '=https' --tlsv1.2 -LsSf \
  https://github.com/axodotdev/cargo-dist/releases/download/v0.32.0/cargo-dist-installer.sh | sh
```

`dist` also keeps a `cargo dist` alias.

## Cutting a release

```bash
# 1. Sanity-check config and planned artifacts
bash scripts/build/check-dist.sh

# 2. Release (example: 0.1.1)
bash scripts/build/release.sh 0.1.1
```

`release.sh` enforces a clean working tree, validates the version format, refuses an
existing tag, bumps `[workspace.package].version`, commits, tags `v0.1.1`, and pushes.
Pushing the tag triggers the workflow, which builds all three platforms, creates the
GitHub Release, and pushes the updated formula to `dfbb/homebrew-tap`.

Track progress at <https://github.com/dfbb/KFCode/actions>.

## Local packaging (no publish)

```bash
bash scripts/build/package-local.sh
```

Builds the **current host target only** (it does not attempt cross-compilation) and
writes the archive plus its `.sha256` to `target/distrib/`. Use it to confirm the
binary packages and runs before tagging a real release.

## How users install

```bash
# Homebrew (macOS arm64 / Linux amd64)
brew install dfbb/tap/kfcode

# shell installer (macOS / Linux)
curl --proto '=https' --tlsv1.2 -LsSf \
  https://github.com/dfbb/KFCode/releases/latest/download/kfcode-cli-installer.sh | sh

# PowerShell installer (Windows)
irm https://github.com/dfbb/KFCode/releases/latest/download/kfcode-cli-installer.ps1 | iex
```

## Maintenance

**Changing the build config.** Edit `dist-workspace.toml` (or the relevant
`Cargo.toml`), then regenerate and verify the workflow:

```bash
dist generate            # rewrites .github/workflows/release.yml
dist generate --check    # CI runs this; fails if the workflow is stale
```

Commit the regenerated `release.yml` together with the config change. CI's
`--check` step fails the build if they drift apart.

**Upgrading dist.** Install the newer version locally, then run `dist init` (or
`dist selfupdate`) to migrate `dist-workspace.toml` and bump `cargo-dist-version`,
and commit the regenerated workflow.

## Known constraints

- **Linux uses glibc (gnu), not musl.** Archives are dynamically linked, so very old
  distros may not run them. This matches the "amd64 tar.gz" requirement; switching to
  musl would complicate the sqlite/openssl builds.
- **`reqwest` default TLS.** On Linux the binary dynamically links `libssl`. The ubuntu
  runner ships `libssl-dev`, so it builds and runs; switching to rustls is out of scope
  for the release pipeline.
- **macOS arm64 only**, as noted above — Intel Macs are not covered.
